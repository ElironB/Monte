# **1\. Product Overview**

## **1.1 What Is Monte Engine?**

Monte Engine is an open-source, self-hostable probabilistic life simulation platform. It ingests deep multi-source personal data about a user, constructs a structured Personal Knowledge Graph of that user, generates 1,000 behavioral clone variants, and runs those clones through configurable decision scenarios - returning probability distributions over outcomes rather than single-point predictions.

The core philosophical position: this tool does not predict the future. It stress-tests a specific human's behavioral tendencies against empirically-grounded world models and returns honest probability distributions. It is a flight simulator for life decisions, not an oracle.

## **1.2 Primary Use Cases**

- Individual decision support: Should I quit my job and day-trade? Start this business? Move cities? Take this degree?
- LLM agent decision layer: External AI agents call the Monte Engine via CLI/API instead of relying on extended internal reasoning, getting empirically-grounded simulation results for a specific user
- Autonomous agent safety: Before an AI agent takes a high-stakes action on behalf of a user, it first runs a simulation to understand the probability distribution of outcomes for this specific user
- Research: Understanding behavioral patterns at scale across anonymized user populations

## **1.3 Why This Architecture Exists**

Current LLMs reasoning about personal decisions have a critical flaw: they reason from generic population-level priors, not from the specific behavioral fingerprint of the individual. A person who has failed to execute 7 budget plans, rage-quits during market volatility, and searches obsessively before any decision is categorically different from someone with identical stated preferences. Monte Engine captures that distinction through revealed behavioral data rather than self-reporting.

Additionally, LLM extended thinking is expensive, slow, and non-reproducible for the same prompt. Monte Engine replaces extended thinking for decision scenarios with structured simulation - cheaper, faster, and returns distributions rather than point estimates.

# **3\. Data Ingestion Layer**

## **3.1 Integration Strategy - Composio**

All third-party data source connections are handled through the Composio SDK. Composio provides 250+ pre-built integrations with managed OAuth, token refresh, and API abstraction. This applies to both the self-hosted and cloud versions - there is no scenario where building custom OAuth connectors makes sense when Composio already maintains them.

The Monte Engine treats Composio as a data transport layer only. Composio fetches the raw data; Monte's signal extractors then process it. The two concerns are fully decoupled - adding a new data source in the future requires writing a signal extractor, not an OAuth connector.

// Composio integration pattern

import { Composio } from '@composio/sdk';

const composio = new Composio({ apiKey: process.env.COMPOSIO_API_KEY });

// Connect a user's data source (returns OAuth URL)

const connectionRequest = await composio.initiateConnection({

userId: userId,

appName: 'REDDIT', // or GOOGLE, NOTION, SPOTIFY, TWITTER, PLAID, etc.

});

// Fetch data via Composio action

const rawData = await composio.executeAction({

action: 'REDDIT_GET_USER_POSTS_AND_COMMENTS',

userId: userId,

params: { limit: 1000, timeRange: '1year' }

});

## **3.4 File Upload Processing (Non-Composio)**

Obsidian vaults (.zip) are uploaded directly. The system extracts all .md files, parses internal wiki links to reconstruct graph structure, and treats link density as a cognitive architecture signal. For images, Groq's vision-capable model extracts lifestyle and context signals. Raw files are deleted after extraction (default: 48 hour retention).

## **3.5 Ingestion Queue Architecture**

All ingestion operations are async and processed through BullMQ. Each source fetch is an independent job. Signal extraction for each source is a child job triggered upon successful fetch completion. The persona build job is triggered only when all source extraction jobs for a user have completed (or timed out with partial data).

Queues:

\- source-fetch (priority: normal, concurrency: 10/user)

\- signal-extraction (priority: high, concurrency: 20 global)

\- persona-build (priority: normal, concurrency: 5 global)

\- simulation-run (priority: configurable, concurrency: 10 global)

Job Lifecycle:

PENDING → ACTIVE → COMPLETED | FAILED

Failed jobs: 3 retries with exponential backoff

Stale jobs (>30min active): auto-fail with partial-data flag

# **4\. Signal Extraction Layer**

## **4.1 The Signal Extraction Philosophy**

This is the most critical layer. Bad extraction produces hallucinated personas. The rule: every signal must be derived from behavioral evidence, never from self-reported content. A user writing 'I am disciplined' in their notes is not a discipline signal. A user with 847 unfinished notes and 12 'budget plan vX' files is a clear execution gap signal.

## **4.2 Per-Source Extractors**

### **4.2.1 Search History Extractor**

- Temporal clustering: Group searches by hour-of-day and day-of-week to identify anxiety windows (3am searches), work patterns, and decision timing
- Repetition analysis: Count repeat searches on same topic within 30-day windows. >5 repeats on same topic = decision paralysis signal on that domain
- Topic taxonomy: LLM-classify search topics into domain clusters (finance, health, relationships, career, entertainment). Distribution reveals true attention allocation
- Threat vs opportunity ratio: Classify each search as threat-motivated ('how to avoid X', 'is X dangerous') vs opportunity-motivated ('how to get X', 'best X for Y'). Ratio is a core anxiety/optimism dimension
- Completion signals: Cross-reference searches with subsequent YouTube views or bookmarks where possible to detect whether curiosity was satisfied or abandoned

### **4.2.2 YouTube Watch History Extractor**

- Completion rate estimation: Derive from rewatch patterns and playlist behavior. Users who rewatch first 2 minutes of many videos = short attention span signal
- Topic depth analysis: Does the user watch introductory content repeatedly on same topics (breadth seeker) or progressively advanced content in a single domain (depth seeker)?
- Emotional regulation pattern: Detect comedy/entertainment spikes following heavy work-topic consumption. Maps to coping style
- Time-of-day consumption: Morning vs evening consumption types reveal routine structure

### **4.2.3 Reddit Behavioral Extractor**

- Upvote vs comment divergence: Topics they silently upvote but never comment on = genuine private interests. Topics they comment on = curated public identity. The gap is the most honest signal Reddit provides
- Subreddit portfolio analysis: Map subreddit mix to personality taxonomy. Presence in r/loseit + r/personalfinance + r/GetMotivated simultaneously signals aspiration-execution gap
- Comment sentiment analysis: Tone in comments (supportive, contrarian, analytical, emotional) mapped per topic domain
- Post timing: Night-posting vs day-posting patterns. Long posts at unusual hours = rumination signal

### **4.2.4 Financial Behavior Extractor (Plaid)**

- Impulse purchase detection: Transactions categorized as discretionary appearing within 48 hours of paycheck deposit, or after-midnight transactions in discretionary categories
- Savings consistency: Month-over-month variance in savings-category transactions. High variance = intention-behavior gap in financial domain
- Subscription accumulation: Count of active subscriptions vs active usage (cross-referenced with Spotify, streaming data where available)
- Emergency fund proxy: Minimum balance patterns over rolling 6-month window
- Stated vs revealed risk: Compare investment account transaction frequency and category to self-reported risk tolerance

### **4.2.5 Cognitive Structure Extractor (Obsidian/Notion)**

- Completion rate: Ratio of notes with clear conclusion vs notes that end mid-thought. Primary execution gap indicator
- Link density: Average internal links per note. High density = networked thinking. Low density = linear thinking
- Versioning signals: Files named with v2, v3, revisedFinal, final_FINAL indicate domains where the user plans extensively but re-plans rather than executes
- Tag usage patterns: Hierarchical tags indicate systematic thinking; no tags indicate impulsive capture
- Time gap analysis: Distribution of time between note creation and last edit. Many notes never opened after creation = cognitive hoarding behavior

## **4.3 Behavioral Dimensions Output Schema**

All extractors converge to a standardized BehavioralDimensions object:

interface BehavioralDimensions {

// Core psychological axes (all -1.0 to 1.0, derived not stated)

temporalOrientation: number; // -1=present-focused, +1=future-focused

lossAversionCoefficient: number; // -1=risk-seeking, +1=highly loss-averse

executionGap: number; // 0=plans=actions, 1=all planning no action

socialProofDependence: number; // 0=independent, 1=requires validation

attentionDepth: number; // -1=breadth seeker, +1=depth specialist

anxietyIndex: number; // 0=low baseline anxiety, 1=high baseline

optimismBias: number; // -1=pessimism bias, +1=optimism bias

cognitiveStyle: 'linear' | 'networked' | 'mixed';

decisionSpeed: number; // 0=fast/impulsive, 1=slow/deliberate

// Domain-specific competence scores (0=novice, 1=expert)

domainCompetence: Record&lt;string, number&gt;;

// Revealed interests (distinct from stated interests)

revealedInterests: Array&lt;{ domain: string; evidence: string; weight: number }&gt;;

// Contradiction registry

contradictions: Array<{

statedBehavior: string;

revealedBehavior: string;

confidence: number;

sources: string\[\];

}>;

// Raw confidence scores per dimension (how much data backed each)

confidenceScores: Record&lt;keyof BehavioralDimensions, number&gt;;

}

# **5\. Persona Construction Layer**

## **5.1 Contradiction Detection**

Before building the knowledge graph, the system runs contradiction detection across all extracted dimensions. Contradictions are first-class data, not noise to be resolved.

Algorithm: For each behavioral dimension, compare values derived from different sources. If variance across sources exceeds 0.3 on normalized scale, flag as a contradiction. Store both values with their source provenance. The contradiction itself becomes a high-weight graph edge.

Example: Reddit reveals high financial anxiety (searches r/personalfinance, upvotes debt-payoff posts). Plaid reveals consistent impulse spending. Both are true simultaneously and the contradiction is the most predictive signal - this user is intellectually engaged with financial wellness but behaviorally unable to execute it. The clone distribution must reflect this split, not average it away.

## **5.4 Clone Generation**

The clone generator creates 1,000 parameter variants from the Master Persona using stratified sampling. Clones are lightweight diffs - not full persona re-generations. The base persona is shared; only the variation delta is stored per clone.

interface CloneVariant {

cloneId: string;

simulationId: string;

basedOnPersonaVersion: int;

// Parameter deltas from base (small floats, positive or negative)

dimensionDeltas: Partial&lt;BehavioralDimensions&gt;;

// Sampling metadata

samplingPercentile: number; // Which part of distribution this clone represents

stratificationGroup: string; // e.g. 'high-anxiety-low-execution'

}

// Stratification groups ensure distribution is meaningful, not random noise:

// 25% clones: base dimensions (near-zero deltas)

// 25% clones: amplified primary dimension (e.g. higher loss aversion)

// 25% clones: suppressed primary dimension

// 25% clones: contradictions resolved one way or another

# **6\. Simulation Engine**

## **6.1 Architecture Principles**

The simulation engine is NOT an LLM generating a story. It is a decision graph traversal system where most computation is statistical, and LLM calls fire only at designated fork points. This is how cost and latency stay tractable at 1,000 clone scale.

| **Component**      | **What It Is**                                                   | **LLM Involved?**               |
| ------------------ | ---------------------------------------------------------------- | ------------------------------- |
| Decision Graph     | Pre-built scenario tree of choice nodes and outcome branches     | No - deterministic structure    |
| World Agents       | Statistical models of external reality (market, employers, etc.) | No - empirical data only        |
| Fork Evaluator     | Evaluates how THIS clone would choose at a decision fork         | Yes - fires once per major fork |
| Chaos Injector     | Randomly inserts black swan events per probability table         | No - random + statistical       |
| Batch Orchestrator | Parallelizes 1,000 clone runs across worker pool                 | No - infrastructure layer       |
| Result Aggregator  | Collects all clone outcomes into probability distributions       | No - statistical aggregation    |

## **6.2 Decision Graph**

A Decision Graph is a directed acyclic graph representing the scenario being simulated. Each node is either a Decision Node (a fork where clone behavior determines path) or an Outcome Node (a terminal state). Edges carry probability weights assigned by World Agents.

interface DecisionGraph {

scenarioId: string;

name: string;

description: string;

timeframeMonths: number;

nodes: DecisionNode\[\];

edges: DecisionEdge\[\];

rootNodeId: string;

terminalNodeIds: string\[\];

}

interface DecisionNode {

id: string;

type: 'decision' | 'outcome' | 'event';

description: string;

worldAgentType?: string; // Which world model governs this node

isMajorFork: boolean; // If true, LLM fork evaluator fires here

month: number; // When in the timeline this node occurs

}

interface DecisionEdge {

fromNodeId: string;

toNodeId: string;

label: string;

basePopulationProbability: number; // From world agent empirical data

}

## **6.3 World Agents**

World Agents represent external reality. They are never LLM-driven. They enforce real-world statistics as hard constraints on simulation outcomes regardless of clone behavior. Each World Agent implements the following interface:

interface WorldAgent {

agentType: string;

// Given a persona and a world state, return probability distribution over outcomes

getOutcomeProbabilities(

persona: MasterPersona,

clone: CloneVariant,

currentState: WorldState,

node: DecisionNode

): OutcomeProbabilityMap;

// Apply a black swan event to world state

applyEvent(event: BlackSwanEvent, state: WorldState): WorldState;

}

### **6.3.1 Financial World Agent - Empirical Data Requirements**

- Day trading: 90% capital ruin within 12 months (retail trader empirical data), 8% break-even, 2% profitable - enforced as hard probability floor
- Startup survival: 90% fail within 10 years, 50% within 5 years - industry-specific variance applied
- Real estate appreciation: Regional market data with 10-year trailing averages
- Index investing: Historical S&P 500 distribution with drawdown data
- The World Agent modulates these base rates based on clone's financial competence score from persona - but cannot violate the base rate by more than 2 standard deviations

### **6.3.2 Career World Agent - Empirical Data Requirements**

- Salary negotiation success rates by domain and experience level
- Remote work availability by industry sector (post-2023 data)
- Job loss probability by economic sector (recession and non-recession conditions)
- Career switch success rates by age cohort and destination domain

### **6.3.3 Education World Agent**

- Degree ROI by field (median salary premium vs cost, 10-year window)
- Completion rates by institution type and age of student
- Certification vs degree outcome comparisons by domain

## **6.4 Fork Evaluator - Dual LLM Provider Strategy**

The Fork Evaluator uses two LLM providers with different roles. Groq handles the bulk of fork evaluations - it runs Llama 3 / Mixtral at ~500 tokens/second with a generous free tier, making the 1,000-clone simulation economically viable and fast. Claude (Anthropic) is reserved for complex fork points where behavioral nuance genuinely requires stronger reasoning, or for persona construction where depth matters more than speed.

| **Task**                                       | **Provider**        | **Rationale**                                                         | **Approx Cost**                  |
| ---------------------------------------------- | ------------------- | --------------------------------------------------------------------- | -------------------------------- |
| Bulk fork evaluation (clones)                  | Groq (Llama 3 70B)  | Fast, free tier, sufficient for behavioral parameter-driven decisions | ~\$0 on free tier                |
| Complex fork evaluation (major life decisions) | Claude Sonnet       | Higher reasoning quality for nuanced scenarios                        | Bounded to < 20 calls/simulation |
| Persona construction                           | Claude Sonnet       | Extracting contradiction patterns, compressing knowledge graph        | Once per persona build           |
| Image signal extraction                        | Groq (vision model) | Fast vision analysis for lifestyle signals                            | Per uploaded image               |
| Quick estimate endpoint                        | Groq (Llama 3 8B)   | Realtime response, ultra-low latency                                  | Effectively free                 |

The LLM router selects provider based on fork complexity score. Forks with complexity &lt; 0.6 (most clone decisions) route to Groq. Forks with complexity &gt;= 0.6 or flagged as isMajorFork=true and involving high-contradiction persona areas route to Claude. This means a typical 1,000-clone simulation uses Groq for ~80-85 calls and Claude for ~10-15 calls.

// Fork Evaluator - provider routing

function selectProvider(fork: DecisionNode, clone: CloneVariant): LLMProvider {

const complexityScore = calculateForkComplexity(fork, clone);

const hasHighContradiction = clone.contradictionMagnitude > 0.6;

if (fork.isMajorFork && hasHighContradiction) return 'anthropic';

if (complexityScore >= 0.6) return 'anthropic';

return 'groq'; // Default - fast, free, sufficient

}

// Fork Evaluator prompt (same structure for both providers)

// System: Behavioral prediction engine. Return JSON only. No prose.

interface ForkEvaluatorInput {

llmContextSummary: string; // Master Persona summary (< 2000 tokens)

cloneDeltaSummary: string; // How this clone differs from base

decisionContext: string; // Fork description

availableChoices: string\[\];

worldStateContext: string;

timeElapsed: number;

}

interface ForkEvaluatorOutput {

choiceProbabilities: Record&lt;string, number&gt;; // Must sum to 1.0

dominantFactors: string\[\];

}

## **6.5 Chaos Injector**

The Chaos Injector inserts Black Swan events stochastically into simulation timelines. These are external events outside the clone's control. Probability tables are configurable per scenario type:

Black Swan Event Library (base probabilities, per 12-month window):

\- Market crash (>30% drawdown): 8% probability

\- Health crisis (significant): 3% probability

\- Relationship disruption (major): 7% probability

\- Unexpected job loss: 10% probability

\- Regulatory change affecting scenario: 5% probability

\- Global pandemic / major disruption: 2% probability

\- Windfall / unexpected positive event: 4% probability

## **6.6 Batch Orchestrator**

The Batch Orchestrator runs 1,000 clones in parallel using the worker pool. Clone runs are stateless - each clone carries its full state through the decision graph without shared mutable state. This enables horizontal scaling.

Orchestration strategy:

\- 1,000 clones split across worker pool (default: 20 workers)

\- Each worker processes 50 clones sequentially

\- Fork evaluator calls are batched: similar clones at same fork share evaluation

\- Cache layer: if two clones are within 0.05 delta of each other on all dimensions,

they share the fork evaluation result (deduplicated LLM calls)

\- Progress events emitted per 10% completion to API layer

\- Target: 1,000 clone simulation completes in < 90 seconds

# **9\. Data Architecture & Privacy**

## **9.1 Data Residency**

For self-hosted deployments: all data stays within the user's own infrastructure. Neo4j, Redis, and MinIO all run locally via Docker Compose. No data leaves the host. For cloud deployments: data is tenant-isolated with separate Neo4j databases per user (not shared graph).

## **9.2 Sensitive Data Handling**

- Raw source data (search history, financial transactions) is stored encrypted at rest in MinIO and deleted after signal extraction completes (default: 48 hours retention, configurable)
- Only derived signals and the knowledge graph persist long-term
- Plaid credentials are never stored - only the transaction data via Plaid's tokenized access
- Images are processed and deleted after vision extraction - pixel data is never stored long-term
- Full GDPR delete endpoint removes all Neo4j nodes, Redis keys, and MinIO objects for a user

## **9.3 Persona Versioning**

Every time new data is ingested and the persona is rebuilt, a new version is created. Previous persona versions are retained for 90 days. Simulations record which persona version they ran against for reproducibility.

### Final Verdict and Go-To-Market Recommendations
The analysis overwhelmingly supports the conclusion that there is a massive, highly capitalized market for probabilistic behavioral simulation, and the Monte Engine should absolutely be built. The transition toward autonomous AI agents necessitates the creation of sophisticated decision-evaluation environments. The Monte Engine's proposed architecture—specifically its combination of Neo4j Knowledge Graphs, contradiction mapping, and Monte Carlo clone generation—represents a mathematically rigorous approach that vastly outperforms existing vector-based competitors.

However, the systemic risks associated with data ingestion APIs and psychological privacy are severe. To ensure commercial success and platform stability, the following strategic recommendations must be integrated into the execution plan:

### Pivot the Data Ingestion Strategy:
Relying on continuous OAuth syncing with major tech platforms is an existential risk due to rapid API volatility and anti-scraping measures. The platform must pivot toward a "Bring Your Own Data" (BYOD) model. Emphasize the file upload extractors—specifically targeting Obsidian vaults, Notion exports, PDF document dumps, and manual Google Takeout archives. Users seeking accurate life simulations are highly motivated and will perform manual data dumps, eliminating reliance on fragile, rate-limited OAuth connections.

### Prioritize the Agentic API over the B2C Interface:
The consumer market for life visualization is currently being contested by beautifully designed, UI-heavy applications. Competing on frontend visualization requires massive resource allocation. Conversely, the monte Engine's true competitive moat lies in its backend rigor. The platform should be positioned as an infrastructure layer for the "Agentic Web." By positioning the CLI and API as tool-calling endpoints for external agent frameworks (e.g., LangChain or proprietary enterprise networks), monte establishes itself as a foundational utility for AI safety and decision evaluation.

### Expand the World Agent Statistical Libraries:
The accuracy of the Monte Carlo simulation depends entirely on the empirical grounding of the World Agents. True competitive differentiation will require vast repositories of actuarial, economic, and sociological base rates. Partnering with data providers or integrating heavily vetted open-source datasets to populate the statistical constraints of these World Agents will be critical to maintaining the simulation's fidelity.
